#Active chat as a state machine

1.Step of making an ACTIVECHAT
On first mount a use effect runs sending request to fetch RECENT CHATROOMS, all existing chatrooms.
One CHATROOM contains:
1.Other user id.
2.Other username.
3.Chat room id.
4.Last message.
5.Time of the last message.
6.Activity status 
On received chat the user id is taken for fetching the profile picture.

ACTIVECHAT with all fields can be be obtained directly by:
1. Selecting a chat existing chat from LEFTSIDEBAR.
2. Writing the username and matching the chat from the LEFTSIDEBAR.
3. From ACTIVECHAT with initialFetchDone false on receiving the initial fetch.
4. Sending a message.
5. Receiving a message.


**Full ACTIVECHAT app state**
The LEFTSIDEBAR is consuming:
Other user id, other username, last messages, activity status, chatroom id as key in the list.

The CHATAREA is consuming:
Other user id, other username, activity status.

THE CHATCONVERSATION is consuming only if ACTIVECHAT is true:
Other username, avatar derived from messages other user id

The INPUTBAR is consuming:
Other username, chat room id


** Search bar selecting a chat **
*Deciding the trigger point that decides the type of the ACTIVECHAT*
ACTIVECHAT can be set also just with the username of the peer.
Meaning if there is already existing chat from RECENT CHATROOMS response, that chat is going to be set as it.
But if there is no chat with that username, new existing chat is opened and now is Active. - So with this design the source of truth is RECENT CHATROOMS response.

Caveates of this design is that RECENT CHATROOMS response can return:
-malformed response not the full list
-returned nothing/network problem

That means the user opening a chat that already exists can get new existing chat which is not.
The source of true can also be on fetching initial if it returns 0 message - no existing chat with that peer.

The caveats besides of network and server responses, here are that the chat is not known which type is it, existing or none existing before the initial response arrives.
*In this frame from opening to  receiving first initial i need to think about what are the user capabilities at that moment! - sending messages is not one of them

Later the RECENT CHATROOM response is going to return like the pagination meaning at one moment all of the chats are not going to be at the client side, so for that reason the more reliable desicion here is the initial fetch to decide the type.
*CONCLUSION the source of the truth needs to be initial, but the source first is decided on the recent chat the failure is decided ultimatly on initial fetch*



So the flow is going to be something like this:
User writes selects a chat based on the username of the peer, first thing is checked does the username === chat.username from the chat list of the sidebar (if it is then ACTIVECHAT is the chat of the list)
If  username !== chat.username we do not conclude decided from the above written the type of the ACTIVE CHAT, instead our knownledge will come with initial fetch.
The ACTIVECHAT with only username sets initialFetchDone false (loading UI) -> initial fetch either moves the UI to existing ACTIVECHAT if there are messages, or new ACTIVECHAT (same peer fields while initialFetchDone was false).
That loading phase ends when initialFetchDone becomes true; ACTIVECHAT can become fully existing via message send/receive or by selecting another chat.



ACTIVECHAT is driving the whole applications:
From the derived info when the ACTIVECHAT is set the components with in are grouped in: 
-UI concerns (activity status, other username(always contained), other user id(avatar)) 
-Logic (chat room id (sending mechanism of messages, forming the recent chat with the key === chat room id))

The app can be viewed as in stages of a different phases of the ACTIVECHAT:
- No ACTIVECHAT
- ACTIVECHAT set only with username / initialFetchDone false (loading) / new chat
- ACTIVECHAT with all fields set / received from the first fetch / sending message / receiving message / with chat selection

**Uploading message image**

Uploading/sending image is like sending message this constrains are: 
-Init phase is finished
-Chat is existing (can not be first message with server design now)

1. Start: taking from the OS system.

The first step happens in INPUTBAR when the photo button is clicked:

**Pre-network phase (local / client-only)** — everything below stays in the browser until upload; no server round-trip yet.
- The OS file picker opens.
- The user chooses a file; it is attached to state `pendingImage` (the image is loaded with asking for sending).
- The app then creates a preview, which is basically a URL attached to the optimistic bubble (the message component uses the URL to display the preview).
- The URL keeps the reference to the bytes in memory (`blob:` URL / `URL.createObjectURL`).

**Network phase (upload / first send)** — the picked `File` leaves the client.
- On submit, the picked file is sent (`File` is metadata for the picked bytes stored in the browser).

Note: that line is the network step; the heading already says network phase. what the request looks like, route, response, errors is not written here - later in upload API description.

- The network phase can return two results.
- First is failure: remove the URL, remove the optimistic message, show an error.
- Second is success:
  - the file is stored in media storage (temp then permanent);
  - the chat server side finishes and the message is delivered to the peer user; the client gets acknowledgement back (MESSAGE_ACK /MESSAGE_RESPONSE) and the optimistic bubble is replaced                           
    with the real message (server id, mediaId).
